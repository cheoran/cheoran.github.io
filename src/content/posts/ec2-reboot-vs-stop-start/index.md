:::note[섹션 개요]

- 재부팅과 종료 후 시작의 동작 차이를 정리한다.
- 왜 stop/start가 SSH 문제를 해결하는지 이해한다.
  :::

인스턴스가 비정상 상태여서 재부팅을 했는데 SSH가 계속 안 붙었다.
그래서 종료 후 시작(Stop/Start)하니 접속이 됐다. 이 차이를 정리해둔다.

## 차이 한눈에 보기

| 항목        | 재부팅(Reboot)            | 종료 후 시작(Stop/Start)                 |
| ----------- | ------------------------- | ---------------------------------------- |
| 호스트      | 같은 호스트에서 OS 재시작 | 새 호스트로 이동(대부분)                 |
| Public IPv4 | 유지                      | 새로 할당(Elastic IP 예외)               |
| 데이터      | 인스턴스 스토어 유지      | RAM + 인스턴스 스토어는 소실, EBS는 유지 |
| 가능 대상   | 대부분                    | EBS 루트 볼륨 인스턴스만 가능            |
| 과금        | 새 과금 사이클 없음       | 다시 시작 시 1분 최소 과금 시작          |

## 왜 재부팅은 안 되고 stop/start는 됐을까

가능성이 큰 흐름은 두 가지다.

1. **호스트 문제**
   재부팅은 같은 호스트에서 OS만 다시 올린다.
   반면 stop/start는 대개 새 호스트로 이동한다.
   즉, 호스트/네트워크 쪽 문제가 있었으면 stop/start가 해결될 수 있다.

2. **Public IP 변경**
   stop/start는 Public IPv4가 바뀐다.
   이때 기존 IP로 접속하면 당연히 실패한다.
   Elastic IP를 붙이거나, 새 IP로 접속해야 한다.

## 기억해둘 포인트

- stop/start는 EBS 루트 인스턴스에서만 가능하다.
- stop/start하면 RAM과 인스턴스 스토어 데이터는 날아간다.
- Public IP가 바뀌는 걸 항상 염두에 둔다. 고정이 필요하면 Elastic IP를 쓴다.
